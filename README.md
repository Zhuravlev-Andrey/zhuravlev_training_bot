## Архитектура проекта

**Разделение `bot.py` и `db.py`:** принцип единственной ответственности. `bot.py` знает только про диалог, `db.py` — только про хранение. Замена БД или добавление веб-интерфейса затрагивает только один файл.

**Пул соединений** настраивается через `DB_POOL_MIN` / `DB_POOL_MAX` без деплоя. При одном пользователе достаточно `min=1, max=3`; при десятках одновременных пользователей — увеличить `max` до 20–50.

**Масштабирование на много пользователей:**
- Все данные разделены по `user_id` (Telegram user ID, BIGINT, глобально уникален — свой UUID не нужен).
- Пул соединений и ресурсы Railway увеличиваются через переменные окружения и настройки плана.
- При сотнях пользователей: перейти с long-polling на webhook, добавить connection pooler (PgBouncer).

**Перенос данных в другое приложение:** команда `/export` выгружает все данные пользователя в JSON или CSV. Формат JSON содержит полную структуру (шаблоны, сессии, веса); CSV — плоскую таблицу подходов для аналитики в Excel/Google Sheets.

## Функциональные требования

### FR-01. Первый запуск / главный экран

При `/start`:
- Если у пользователя нет тренировок → предложить создать первую.
- Если есть → показать историю текущего месяца и список тренировок.

```
📅 Тренировки в этом месяце:
1. ✅ 01.06 10:30 — Грудь + Трицепс
2. ❌ 03.06 18:15 — Ноги + Плечи (отменена)

Выберите тренировку:
[Грудь + Трицепс]
[Ноги + Плечи]
[Спина + Бицепс]
[➕ Создать новую тренировку]
[📤 Экспорт данных]
```

### FR-02. Создание тренировки

Пользователь проходит диалог:

1. Название тренировки (текст).
2. Включить прогрессию весов? (Да / Нет) — настройка на уровне тренировки.
3. Для каждого упражнения:
   - Название упражнения.
   - Тренажёр: выбрать из существующих в `equipment` или создать новый (ввести название и веса через запятую).
   - Количество подходов (по умолчанию 4).
   - Повторений в подходе (по умолчанию 10, одинаково для всех подходов; переопределяется через `set_template`).
   - Время отдыха в секундах (по умолчанию 90).
   - Комментарий (необязательно, `/skip` для пропуска).
4. Добавить ещё упражнение или завершить.

После создания предложить сразу начать тренировку.

### FR-03. Выполнение тренировки

- Упражнения выполняются последовательно.
- Перед каждым упражнением: название, рабочий вес, дата последнего выполнения, комментарий.
- Для каждого подхода:
  - Показать рекомендуемый вес (±4 от рекомендуемого + кнопка «Все веса»).
  - Если тренажёра нет — ввести вес текстом.
  - Выбрать повторения (1–12).
  - Показать таймер отдыха + кнопка «Пропустить».
- Каждый подход сохраняется в БД немедленно.

### FR-04. Прогрессия весов

- Работает только если включена при создании тренировки (`use_weight_progress = TRUE`).
- Применяется после последнего подхода упражнения.
- Алгоритм (см. раздел [Логика тренировки](#логика-тренировки)).
- При изменении веса — запрос подтверждения.

### FR-05. Завершение и отмена тренировки

- Завершение: сессия получает статус `completed`, финальное сообщение + кнопка «Начать новую».
- Отмена: статус `cancelled`, данные выполненных упражнений сохранены, незавершённое — в лог с `new_max_kg = NULL`.

### FR-06. Таймер отдыха

После каждого подхода (кроме последнего):
```
⏱ Отдых: 1:30
[⏭ Пропустить]
```
Переход к следующему подходу только через кнопку «Пропустить» или по истечении таймера.

### FR-07. Экспорт данных

Команда `/export`:
- JSON: полная структура (шаблоны тренировок, все сессии с подходами, актуальные веса).
- CSV: плоская таблица `дата, тренировка, упражнение, подход, вес, повторения, новый_макс`.
- Файл отправляется в чат как документ.

### FR-08. Расписание (заложено в БД)

Таблица `schedule_entry` хранит привязку тренировки к дням недели и времени напоминания. Реализация команд управления расписанием — следующая версия.

### FR-09. Аналитика (следующая версия)

- Прогресс по упражнению: динамика рабочего веса по датам.
- Объём тренировки: суммарный тоннаж за сессию.
- Частота: количество тренировок по неделям/месяцам.

## Нефункциональные требования

| # | Требование | Метрика / решение |
|---|---|---|
| NFR-01 | Время отклика | ≤ 2 сек при нормальной нагрузке. Telegram long-polling добавляет 1 round-trip по сети — больше чем у веб (1 сек) и мобайла (100 мс) |
| NFR-02 | Надёжность | Каждый подход сохраняется немедленно. Сбой в середине тренировки не теряет данные |
| NFR-03 | Масштабируемость | `DB_POOL_MAX` настраивается без деплоя. При сотнях пользователей — webhook + PgBouncer |
| NFR-04 | Поддерживаемость | Структура тренировок в БД, не в коде. Оборудование в `equipment` — меняется через SQL |
| NFR-05 | Доступность | Long-polling не требует публичного адреса. Railway перезапускает процесс при падении |
| NFR-06 | Переносимость данных | Экспорт JSON + CSV через `/export` |

## Логика тренировки

### Схема подходов (по умолчанию)

Настраивается при создании упражнения. Переопределения на уровне конкретного подхода хранятся в `set_template`.

| Подход | Повторения | Вес | Отдых |
|---|---|---|---|
| 1 | `default_reps` (или override) | `weight_pct` × рабочий | `default_rest_s` (или override) |
| 2 | то же | то же | то же |
| … | … | … | … |
| N | то же | то же | — |

### Алгоритм прогрессии (если включён)

Применяется только если `use_weight_progress = TRUE` и подходов ≥ 4.

```
если подход_3.reps < target_reps:
    → снизить: available_weight[idx - 1]
    (приоритет — даже если подход_4 выполнен)

иначе если подход_4.reps == target_reps
         И подход_4.weight_kg == рабочий_вес:
    → повысить: available_weight[idx + 1]

иначе:
    → сохранить
```

На границах списка (мин/макс) вес не меняется.

## Модель данных

Все таблицы именуются в **единственном числе**.

### `equipment` — снаряд

| Колонка | Тип | Описание |
|---|---|---|
| `id` | SERIAL PK | Автоинкремент |
| `eq_key` | TEXT UNIQUE | Системный ключ: `smith_machine`, `dumbbells` и др. |
| `name` | TEXT | Отображаемое название |
| `available_weight` | REAL[] | Упорядоченный массив весов в кг. Порядок критичен: прогрессия работает через `idx ± 1`. REAL — 32-бит float, точности достаточно для весов |

Начальные данные вставляются через `ON CONFLICT DO NOTHING` при `init_db()`. Изменить веса можно через SQL без деплоя.

### `workout_template` — шаблон тренировки

| Колонка | Тип | Описание |
|---|---|---|
| `id` | SERIAL PK | |
| `user_id` | BIGINT | Telegram user ID. BIGINT (64-бит) — Telegram использует большие числа |
| `name` | TEXT | Название, например «Грудь + Трицепс» |
| `use_weight_progress` | BOOLEAN | Включить прогрессию весов для всей тренировки |
| `schedule_order` | INT | Порядок в расписании (`NULL` = не включена в расписание) |
| `created_at` | TIMESTAMPTZ | Дата создания |

### `exercise_template` — упражнение в шаблоне

| Колонка | Тип | Описание |
|---|---|---|
| `id` | SERIAL PK | |
| `workout_id` | INT FK → `workout_template.id` CASCADE | Принадлежность к тренировке |
| `equipment_id` | INT FK → `equipment.id` SET NULL | Снаряд. `NULL` = без тренажёра (вес вводится вручную) |
| `name` | TEXT | Название упражнения |
| `order_index` | INT | Порядок в тренировке |
| `default_sets` | INT | Количество подходов по умолчанию |
| `default_reps` | INT | Повторений по умолчанию для всех подходов |
| `default_rest_s` | INT | Отдых в секундах по умолчанию |
| `comment` | TEXT | Заметка для пользователя (техника, нюансы) |

### `set_template` — переопределение подхода

Строка создаётся только если подход отличается от дефолтных настроек упражнения.

| Колонка | Тип | Описание |
|---|---|---|
| `id` | SERIAL PK | |
| `exercise_id` | INT FK → `exercise_template.id` CASCADE | |
| `set_number` | INT | Номер подхода (1-based) |
| `reps` | INT | `NULL` = брать из `exercise_template.default_reps` |
| `rest_s` | INT | `NULL` = брать из `exercise_template.default_rest_s` |
| `weight_pct` | REAL | Доля от рабочего веса (например 0.4, 0.7). `NULL` = 1.0 (100%) |

Уникальный ключ: `(exercise_id, set_number)`.

### `training_weight` — актуальный рабочий вес

| Колонка | Тип | Описание |
|---|---|---|
| `user_id` | BIGINT PK | |
| `exercise_id` | INT PK FK → `exercise_template.id` CASCADE | |
| `weight_kg` | REAL | Текущий рабочий вес |
| `updated_at` | TIMESTAMPTZ | Дата последнего обновления |

Составной PK: `(user_id, exercise_id)`. При первом обращении запись создаётся с весом, который пользователь указал при первом выполнении упражнения.

### `workout_session` — факт тренировки

| Колонка | Тип | Описание |
|---|---|---|
| `id` | SERIAL PK | |
| `user_id` | BIGINT | |
| `template_id` | INT FK → `workout_template.id` SET NULL | `NULL` если шаблон удалён |
| `template_name` | TEXT | Денормализовано: название на момент выполнения |
| `status` | TEXT | `active` / `completed` / `cancelled` |
| `started_at` | TIMESTAMPTZ | |
| `finished_at` | TIMESTAMPTZ | `NULL` пока тренировка не завершена |

`template_name` денормализован: если пользователь переименует тренировку, история останется читаемой.

### `workout_set` — выполненный подход

| Колонка | Тип | Описание |
|---|---|---|
| `id` | SERIAL PK | |
| `session_id` | INT FK → `workout_session.id` CASCADE | |
| `exercise_id` | INT | ID упражнения |
| `exercise_name` | TEXT | Денормализовано (справочник в коде) |
| `set_number` | INT | Номер подхода |
| `weight_kg` | REAL | Фактический вес |

### `schedule_entry` — расписание

| Колонка | Тип | Описание |
|---|---|---|
| `id` | SERIAL PK | |
| `user_id` | BIGINT | |
| `template_id` | INT FK → `workout_template.id` CASCADE | |
| `weekday` | SMALLINT | 0=пн … 6=вс. `NULL` = без привязки к дню |
| `remind_time` | TIME | Время напоминания. `NULL` = без напоминания |

## Связи между таблицами

```
equipment (id)
    ↑ FK от exercise_template.equipment_id (SET NULL при удалении)

workout_template (id)
    ├── exercise_template (workout_id) ON DELETE CASCADE
    │       ├── set_template (exercise_id) ON DELETE CASCADE
    │       └── training_weight (exercise_id) ON DELETE CASCADE
    ├── workout_session (template_id) ON DELETE SET NULL
    └── schedule_entry (template_id) ON DELETE CASCADE

workout_session (id)
    └── workout_set (session_id) ON DELETE CASCADE
```

**SET NULL vs CASCADE:**
- `workout_session.template_id` → SET NULL: удаление шаблона не удаляет историю выполнений.
- `exercise_template` → CASCADE: удаление тренировки удаляет все её упражнения, переопределения подходов и рабочие веса.
- `workout_set` → CASCADE: удаление сессии удаляет все её подходы.

## Диаграмма FSM

FSM (Finite State Machine) — бот всегда в одном состоянии и реагирует только на допустимые для него действия.

```
/start
    │
    ├─ [нет тренировок] → предложить создать
    │
    └─ [есть тренировки] → WRK_SELECT
            │
            ├─ new_workout → CRT_NAME → CRT_PROGRESS → CRT_EX_NAME
            │       → CRT_EX_EQUIP → [eq_new → CRT_EX_EQUIP_NEW_NAME
            │                                 → CRT_EX_EQUIP_NEW_WEIGHTS]
            │       → CRT_EX_SETS → CRT_EX_REPS → CRT_EX_REST
            │       → CRT_EX_COMMENT → CRT_EX_MORE
            │               ├─ [ещё упражнение] → CRT_EX_NAME
            │               └─ [готово] → сохранение → WRK_SELECT
            │
            └─ tmpl_N → создание сессии
                    │
                    └─ [для каждого упражнения]
                            │
                            ▼
                        WRK_WEIGHT ◄────────────────────────┐
                            │ w_* / текст                    │
                            ▼                                │
                        WRK_REPS                             │
                            │ r_*                            │
                            ▼                                │
                        WRK_REST                             │
                            │ skip_rest                      │
                            │                    [подход < N]│
                            └────────────────────────────────┘
                            [последний подход]
                            │
                            ▼
                    finish_exercise()
                            │
                            ├─ [вес меняется] → WRK_CONFIRM_CHANGE
                            │                       │ wchange_yes/no
                            │                       ▼
                            └─ [вес не меняется] → _apply_exercise_result()
                                                        │
                                        ┌───────────────┘
                                        ├─ [ещё упражнения] → WRK_WEIGHT
                                        └─ [все готово] → END

cancel_workout → сессия cancelled → END
/cancel → END (в любом состоянии)
/export → EXPORT_FORMAT → export_json/csv → END
```

## Требования безопасности

| # | Требование | Реализация |
|---|---|---|
| SEC-01 | Токен бота | Только в `TOKEN` env; не в коде и не в логах |
| SEC-02 | Изоляция пользователей | Все запросы фильтруются по `user_id = update.effective_user.id`; подменить нельзя |
| SEC-03 | Владение данными | `get_workout_template(id, user_id)` — проверяет что шаблон принадлежит пользователю |
| SEC-04 | Валидация callback_data | `try/except (IndexError, ValueError)` на все парсинги; ошибка → сообщение, не падение |
| SEC-05 | SQL-инъекции | Только параметризованные запросы asyncpg (`$1`, `$2`, ...) |
| SEC-06 | DATABASE_URL | Только в env; Railway шифрует переменные |
| SEC-07 | Экспорт | Пользователь получает только свои данные (фильтр по `user_id`) |

## Тестирование

### T-01. Юнит: алгоритм прогрессии

| Сценарий | Вход | Ожидание |
|---|---|---|
| Подход 3 < target | set3=8, set4=10 | Вес снижается |
| Подход 3 < target, подход 4 тоже | set3=7, set4=8 | Вес снижается (приоритет) |
| Подход 4 == target с рабочим весом | set3=10, set4=10, set4_w=current | Вес повышается |
| Подход 4 == target с нерабочим весом | set3=10, set4=10, set4_w≠current | Без изменений |
| Подход 4 < target | set3=10, set4=9 | Без изменений |
| Уже максимум | current=max(available) | Без изменений |
| Уже минимум | current=min(available) | Без изменений |
| Прогрессия выключена | use_progress=False | Всегда без изменений |

### T-02. Юнит: выбор ближайшего веса

| Сценарий | Вход | Ожидание |
|---|---|---|
| Точное совпадение | [20,25,30], target=25 | 25 |
| Между двумя | [20,25,30], target=23 | 25 |
| Ниже минимума | [20,25,30], target=5 | 20 |
| Выше максимума | [20,25,30], target=100 | 30 |

### T-03. Интеграция: БД

- `init_db()` идемпотентна (повторный вызов без ошибок и дублей).
- `get_weight()` создаёт запись при первом обращении.
- `set_weight()` корректно делает upsert.
- Удаление `workout_template` → каскадно удаляет `exercise_template`, `set_template`, `training_weight`.
- Удаление `workout_template` → `workout_session.template_id` становится NULL (история сохраняется).
- `get_month_session()` не возвращает прошлый месяц.
- `export_user_data()` возвращает только данные запрошенного `user_id`.

### T-04. Ручные сценарии

| Сценарий | Проверить |
|---|---|
| Первый запуск | Предложение создать тренировку |
| Создание тренировки с новым тренажёром | Новая запись в `equipment`, упражнение связано с ней |
| Создание упражнения без тренажёра | Ввод веса текстом во время тренировки |
| Таймер отдыха | Показывается между подходами; «Пропустить» переходит к следующему |
| Прогрессия выключена | После последнего подхода нет запроса на изменение веса |
| Отмена после 2-го упражнения | Сессия `cancelled`; веса 1-го и 2-го упражнений сохранены |
| `/export` JSON | Файл содержит шаблоны, сессии, веса только текущего пользователя |
| `/export` CSV | Файл открывается в Excel, кириллица читается (UTF-8 BOM) |
| Несколько пользователей | Данные пользователя A не видны пользователю B |

## Известные ограничения и точки роста

| # | Ограничение | Решение |
|---|---|---|
| 1 | Расписание хранится в БД, но команды управления не реализованы | Добавить `/schedule` — привязка тренировки к дням недели |
| 2 | Напоминания не реализованы | APScheduler: фоновая задача читает `schedule_entry` и шлёт сообщения |
| 3 | Аналитика не реализована | `/stats` — динамика весов, объём тренировки, частота |
| 4 | Переопределения подходов (`set_template`) создаются только через SQL | Добавить диалог редактирования подходов внутри бота |
| 5 | При `/start` во время тренировки диалог начинается заново | Спрашивать: «Тренировка уже идёт. Завершить или продолжить?» |
| 6 | Упражнения из справочника (по группам мышц) не реализованы | Добавить таблицу `exercise_catalog` с полями `muscle_group`, `eq_id` |
| 7 | Нет редактирования созданных тренировок | Добавить команду `/edit` |
| 8 | Long-polling не масштабируется на тысячи пользователей | Перейти на webhook + PgBouncer при росте нагрузки |
